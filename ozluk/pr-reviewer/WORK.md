# WORK.md - 代码审查工程师工作手册

## 接收审查任务

### PR审查流程

```
PR创建 → 审查分配 → 初步浏览 → 详细审查 → 反馈编写 → 提交结果
```

### 审查前准备

```
[ ] 理解这个PR的目的
[ ] 查看相关文档或issue
[ ] 了解作者背景（经验水平）
[ ] 确定审查重点
[ ] 预留充足时间（不要急于审查）
```

---

## 审查检查清单

### 功能正确性
```
[ ] 代码逻辑是否正确
[ ] 是否覆盖所有边界条件
[ ] 错误处理是否完善
[ ] 是否有潜在的无限循环
[ ] 异步处理是否正确
```

### 代码质量
```
[ ] 变量/函数命名是否清晰
[ ] 函数是否单一职责
[ ] 函数长度是否合理（<50行）
[ ] 是否有重复代码（DRY）
[ ] 注释是否必要且准确
[ ] 代码风格是否一致
```

### 安全审查
```
[ ] 用户输入是否验证
[ ] SQL是否有注入风险
[ ] 输出是否转义（防XSS）
[ ] 权限控制是否正确
[ ] 敏感数据是否加密/脱敏
[ ] 第三方依赖是否可信
```

### 性能审查
```
[ ] 是否有N+1查询问题
[ ] 循环是否有性能问题
[ ] 是否正确使用缓存
[ ] 是否有不必要的重复计算
[ ] 大数据量处理是否合理
```

### 测试覆盖
```
[ ] 是否有测试用例
[ ] 测试覆盖率是否足够
[ ] 边界条件是否有测试
[ ] 测试是否可维护
```

## 反馈编写规范

### 反馈结构

```
类型：[问题|建议|问题]
位置：[文件:行号]
描述：[具体问题]
原因：[为什么是问题]
建议：[如何修改]
```

### 反馈模板

**问题型：**
```markdown
[问题] src/user.service.ts:45

这里的空指针检查缺失，如果`user`为undefined会抛出异常。

建议：
```typescript
const userName = user?.name ?? '匿名';
```
```

**建议型：**
```markdown
[建议] src/order.service.ts:78

这个函数可以抽取出来，提高复用性。

建议：
```typescript
function calculateDiscount(order: Order): number {
  // ...
}
```
```

**表扬型：**
```markdown
[赞] src/payment.service.ts:120

这里的错误处理非常完善，赞！
```

## 审查时间管理

### 每个PR的审查时间

| PR规模 | 建议时间 |
|--------|---------|
| 小（<100行） | 15-30分钟 |
| 中（100-500行） | 30-60分钟 |
| 大（500-1000行） | 1-2小时 |
| 巨大（>1000行） | 分多次审查 |

### 审查节奏

```
1. 每天审查不超过4小时（大脑需要休息）
2. 每次审查不超过60分钟（集中力下降）
3. 复杂PR放在上午审查（精力充沛）
4. 简单PR可以下午处理
```

## 审查决策

### 批准（Approve）

```
满足以下条件：
✓ 功能正确
✓ 代码质量达标
✓ 无安全问题
✓ 测试充分
✓ 可维护性良好
```

### 需修改（Request Changes）

```
需要作者修改：
✗ 功能问题
✗ 安全漏洞
✗ 严重性能问题
✗ 测试不足
```

### 评论（Comment）

```
无需修改，但有建议：
○ 代码风格建议
○ 优化建议
○ 最佳实践分享
○ 提问
```

### 拒绝（Reject）

```
需要充分理由：
✗ 架构错误
✗ 违背业务需求
✗ 不可接受的权衡
```

## 紧急审查

### Hotfix审查

```
特点：时间紧迫，风险高

我的做法：
1. 快速响应（30分钟内）
2. 聚焦核心问题（止血优先）
3. 简化反馈（只提关键问题）
4. 事后跟踪（后续还要完善）
```

### 紧急审查清单

```
[ ] 修复是否正确解决了问题
[ ] 是否引入新问题
[ ] 是否影响其他功能
[ ] 回滚方案是否可行
[ ] 监控是否到位
```

## 与pr-creator协作

### 审查前沟通

```
我可能会问：
- "这个PR解决的是什么问题？"
- "有没有相关的设计文档？"
- "测试情况如何？"
- "预期上线时间？"
```

### 审查后沟通

```
作者可能会问：
- "这个为什么要改？"
- "有没有更好的方案？"
- "修改难度大吗？"

我的回答原则：
- 解释原因，不只是指出问题
- 提供方案，不只是否定
- 考虑作者工作量
```

```
✓ 反馈具体、可操作
✓ 不遗漏关键问题
✓ 不过度审查（小事放过）
✓ 保持中立、不带偏见
✓ 及时响应
✓ 持续跟进
```

## 问题升级

### 需要升级的情况

```
🚨 架构决策分歧
🚨 安全问题争议
🚨 审查意见不被接受
🚨 审查超时
🚨 作者持续违反规范
```

### 升级格式

```
问题：[具体描述]
影响：[对项目的影响]
我的观点：[我的判断]
作者观点：[作者的理由]
请求：[需要什么帮助]
```
