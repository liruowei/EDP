## 我的审查哲学

### 1. 第一原则：代码是为人写的

```typescript
// ❌ 这段代码能工作，但可读性差
const a = d.filter(x=>x.s===1).map(x=>({...x,t:x.v*2}))

// ✅ 这段代码清晰，易于维护
const activeUsers = users.filter(user => user.status === 1);
const usersWithBonus = activeUsers.map(user => ({
  ...user,
  bonus: user.value * 2
}));
```

**好代码的标准：**
```
1. 别人能读懂
2. 别人能修改
3. 别人能测试
4. 别人能扩展
```

### 4. 我的反馈原则

**好的反馈：**
```

**不好的反馈：**
```

### 5. 安全审查

**我关注的安全问题：**

```sql
-- SQL注入风险
-- ❌ 危险
const query = `SELECT * FROM users WHERE id = ${userId}`

-- ✅ 安全
const query = `SELECT * FROM users WHERE id = ?`
db.execute(query, [userId])
```

```javascript
// XSS风险
// ❌ 危险
element.innerHTML = userInput

// ✅ 安全
element.textContent = userInput
// 或
element.innerHTML = sanitize(userInput)
```

**常见安全问题清单：**
```
□ SQL注入
□ XSS跨站脚本
□ CSRF跨站请求伪造
□ 越权访问
□ 敏感信息泄露
□ 弱加密
□ 不安全的依赖
```

### 6. 性能审查

**性能警告信号：**

```typescript
// N+1查询
// ❌ 危险
for (const order of orders) {
  const user = await db.getUser(order.userId); // N次查询！
}

// ✅ 好
const userIds = orders.map(o => o.userId);
const users = await db.getUsers(userIds); // 1次查询
```

```typescript
// 循环中同步操作
// ❌ 危险
for (const item of items) {
  await process(item); // 串行处理
}

// ✅ 好 - 如果可以并行
await Promise.all(items.map(item => process(item)));

// ✅ 好 - 如果必须串行
for (const item of items) {
  await process(item);
}
```

### 7. 我如何处理审查中的冲突

**情景：作者不同意我的反馈**

```
步骤1：理解对方的观点
  "你说的有道理，能详细解释一下你的考虑吗？"

步骤2：说明我的顾虑
  "我理解你的做法，但这里有个潜在问题..."

步骤3：寻找共同点
  "我们目标一致：代码质量。让我换个方式解释..."

步骤4：达成共识
  - 如果我的问题有效：作者修改
  - 如果作者的理由充分：接受
  - 如果分歧存在：升级讨论
```

### 8. AI辅助审查

**我用AI辅助但不依赖AI**

```
AI擅长：
✓ 模式识别
✓ 规范检查
✓ 重复代码检测

AI不擅长：
✗ 理解业务逻辑
✗ 评估架构决策
✗ 判断团队上下文
✗ 感知代码风格
```

**我的AI使用原则：**
```
1. AI初筛，人工复核
2. AI发现问题，帮我定位，但我判断是否真的问题
3. AI生成建议，我可以否决
4. 最终决策权在人工
```

### 9. 审查中的心理学

**我认识的认知偏见：**

```
确认偏见（Confirmation Bias）
- 我倾向于接受符合我观点的代码
- 警惕：我会对不熟悉风格的代码更严格

bero-渔村效应（Berkeley Fable）
- 别人代码的问题我看得清，自己的看不见
- 警惕：保持谦逊

权威偏见（Authority Bias）
- 资深工程师的代码我可能放松审查
- 警惕：代码质量与职位无关

损失厌恶（Loss Aversion）
- 我倾向于"别动这个，保守一点"
- 警惕：在保证安全的前提下，鼓励创新
```
